# Навык защиты Rust + WASM + HTML проектов

**Supply Chain Defense + Secure Development Skill** для AI-агентов (Claude Code, Cursor и др.)

**Версия:** 1.0 (May 2026)
**Цель:** Защита Rust + WASM + HTML проектов от supply chain атак, небезопасного кода и рисков при работе в режиме vibe coding / agentic coding.

## Основные принципы

1. **Cargo и crates.io** — основной источник рисков supply chain в Rust.
2. **WASM** — дополнительные риски из-за взаимодействия с JavaScript/HTML.
3. **HTML/JS interop** — даже если основная логика на Rust, frontend часть может стать точкой входа.
4. **unsafe** код — минимизировать и контролировать.

## Обязательные правила перед добавлением зависимости (Cargo)

Перед тем, как предлагать `cargo add` или изменять `Cargo.toml`, агент **обязан** проверить:

### Чек-лист проверки крейта
- [ ] Крейт существует на crates.io?
- [ ] Количество загрузок и звёзд (low downloads = риск)
- [ ] Дата последнего обновления
- [ ] Есть ли `build.rs` или посторонние скрипты?
- [ ] Использует ли крейт `unsafe` в публичном API?
- [ ] Есть ли известные уязвимости или компрометации?

** При наличии красных флагов** — явно предупредить пользователя и предложить альтернативу или ручную проверку.

## Правила для WASM + HTML

- При генерации `wasm-bindgen` и `web-sys` — минимизировать использование `js_sys` и прямых `js!` вызовов.
- При работе с Canvas / DOM — предлагать использование typed arrays и безопасных абстракций.
- Не предлагать `innerHTML` / `outerHTML` без санитайзации.
- При использовании Trunk — проверять `Trunk.toml` и настройки безопасности.

## Правила по `unsafe`

- По возможности избегать `unsafe` код.
- Если `unsafe` необходим — обязательно добавлять `// SAFETY:` комментарий с обоснованием.
- При генерации FFI / wasm-bindgen — проверять границы безопасности.

## Правила для vibe coding в Rust

- Не предлагать добавлять крейты "потому что надо быстро".
- При генерации нового проекта — начинать с минимального `Cargo.toml`.
- Предлагать `cargo audit` и `cargo vet` (если используется).
- При работе с WASM — проверять `wasm-opt` и безопасные настройки.

## Реагирование на подозрение в компрометации

Если подозревается компрометация крейта:
1. Не торопиться с `cargo update`.
2. Проверить `Cargo.lock` и историю изменений.
3. Предложить изоляцию окружения перед ротацией токенов.
4. Провести аудит всех зависимых проектов.
